## Key Findings

### 1. Encryption Algorithm: SFrame (Secure Frame)

**Confirmed:** The primary encryption algorithm for Matrix RTC E2EE is **SFrame** (Secure Frame), as defined in IETF draft `draft-ietf-sframe-enc`.

**SFrame Details:**
- **Protocol:** End-to-end encryption for media frames in multiparty conference calls
- **Transport Independent:** Works with RTP, WebTransport, and other protocols
- **SFU Compatible:** Allows media servers to access routing metadata without decrypting content
- **Frame-level Encryption:** Applied to whole media frames rather than individual packets for efficiency

### 2. Key Derivation Function Parameters

From the SFrame specification:

**Base Key Derivation:**
```
sframe_secret = HKDF-Extract("", base_key)
sframe_key_label = "SFrame 1.0 Secret key " + KID + cipher_suite  
sframe_key = HKDF-Expand(sframe_secret, sframe_key_label, AEAD.Nk)
sframe_salt_label = "SFrame 1.0 Secret salt " + KID + cipher_suite
sframe_salt = HKDF-Expand(sframe_secret, sframe_salt_label, AEAD.Nn)
```

**Nonce Generation:**
- Nonce = XOR(sframe_salt, counter)
- Counter encoded as big-endian integer of length AEAD.Nn

**Key Management Integration:**
- Can use MLS (Messaging Layer Security) for group key management
- Or sender keys for simpler scenarios
- KID structure varies by key management scheme

### 3. Cipher Suites Supported

| Cipher Suite | Hash | Key Size | Nonce | Tag Size |
|--------------|------|----------|--------|----------|
| AES_128_CTR_HMAC_SHA256_80 | SHA256 | 48 bytes | 12 bytes | 10 bytes |
| AES_128_CTR_HMAC_SHA256_64 | SHA256 | 48 bytes | 12 bytes | 8 bytes |
| AES_128_CTR_HMAC_SHA256_32 | SHA256 | 48 bytes | 12 bytes | 4 bytes |
| AES_128_GCM_SHA256_128 | SHA256 | 16 bytes | 12 bytes | 16 bytes |
| AES_256_GCM_SHA512_128 | SHA512 | 32 bytes | 12 bytes | 16 bytes |

### 4. Key Exchange Between Participants

**Two Primary Schemes:**

**Option A: Sender Keys**
- Each sender generates own base_key
- Distributed via pre-existing E2E secure channel
- KID format: `(key_generation << R) + (ratchet_step % (1 << R))`
- Forward secrecy via key ratcheting: `base_key[i+1] = HKDF-Expand(HKDF-Extract("", base_key[i]), "SFrame 1.0 Ratchet", CipherSuite.Nh)`

**Option B: MLS Integration**
- Uses MLS (Messaging Layer Security) for group key management
- Shared base_key per epoch: `base_key = MLS-Exporter("SFrame 1.0 Base Key", "", AEAD.Nk)`
- KID format: `(context << (S + E)) + (sender_index << E) + (epoch % (1 << E))`

## Technical Implementation Notes

### SFrame Header Structure
```
 0 1 2 3 4 5 6 7
+-+-+-+-+-+-+-+-+
|X|    KID      |  
+-+-+-+-+-+-+-+-+
|   CTR (opt)   |
+-+-+-+-+-+-+-+-+
```

### Encryption Process
1. Derive sframe_key and sframe_salt from base_key + KID
2. Generate nonce = sframe_salt XOR counter  
3. Encrypt with AEAD: `AEAD.Encrypt(sframe_key, nonce, header + metadata, plaintext)`
4. Return: header + ciphertext

### Security Properties
- **No per-sender authentication** (symmetric encryption)
- **Replay protection** must be handled by application
- **Forward secrecy** via key ratcheting or MLS epoch updates
- **SFU transparency** - media servers can route without decryption

## Conclusion

### Primary Finding: SFrame is the Encryption Algorithm
The research confirms that **SFrame (Secure Frame)** is the core encryption technology for Matrix RTC E2EE integration with LiveKit. SFrame provides:

1. **Transport-agnostic E2EE** suitable for both RTP and newer protocols like WebTransport
2. **SFU compatibility** allowing media servers to route traffic without decrypting content  
3. **Flexible key management** supporting both simple sender keys and advanced MLS group keys
4. **Efficient frame-level encryption** with configurable authentication tag sizes

### Implementation Approach
For any Matrix RTC E2EE implementation:

1. **Use SFrame specification** as the encryption standard
2. **Implement or integrate** SFrame encryption/decryption
3. **Choose key management scheme:**
   - Sender Keys for simpler scenarios
   - MLS integration for advanced group management with forward secrecy
4. **Handle key derivation** using HKDF as specified
5. **Use reference implementation** for testing interoperability

### Missing Pieces for Full Implementation
1. **MSC4143 specification** - need actual Matrix proposal document
2. **Production SFrame libraries** for target languages (Python/TypeScript/etc.)
3. **Matrix-LiveKit integration details** - how the two systems coordinate
4. **Key management implementation** - either sender keys or MLS integration
